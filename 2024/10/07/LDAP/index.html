Java LDAP学习

LDAP

概述

基本了解

Lightweight Directory Access Protocol，轻型目录访问协议，是一种目录服务协议。它以树状结构组织数据，它提供了一种查询、浏览、搜索和修改互联网目录数据的机制，运行在TCP/IP协议栈之上。

LDAP目录服务基于客户端-服务器模型，约定了Client与Server之间的信息交互格式、使用的端口号、认证方式等内容。

OpenLDAP 是可以运行在Linux上的LDAP协议的开源实现。而我们平常说的 LDAP Server，一般指的是安装并配置了OpenLDAP 这些程序的服务器。

Java对象在LDAP目录中也有多种存储形式：

  • Java序列化
  • JNDI Reference
  • Marshalled对象
  • Remote Location

协议运作概览

客户端发送 LDAP 会话连接到 LDAP 服务器。默认连接端口是 389（LDAP），或者 636（LDAPS）。客户端发送操作请求到服务器，服务器返回响应结果。客户端无需等待服务器响应即可发送下一个请求。服务器则会一次次返回响应结果。

Ldap基本模型

每一个系统、协议都会有属于自己的模型，LDAP同样有。在LDAP中，非常重要的就是目录树，下面简单了解一下。

目录树

在LDAP中目录时按照树型结构组织–目录信息树（DIT），DIT是一个主要进行读操作的数据库。

目录树概念

  1. 目录树：在一个目录服务系统中，整个目录信息集可以表示为一个目录信息树，树中的每个节点是一个条目（Entry）。
  2. 条目：每个条目就是一条记录，每个条目都有唯一可区别的名称（DN）。
  3. 对象类：与某个实体类型对应的一组属性，对象类是可以继承的，这样父类的必须属性也会被继承下来（ObjectClass）。
  4. 属性：描述条目的某个方面的信息，一个属性由一个属性类型和一个或多个属性值组成，属性有必须属性和非必须属性（Attribute）

简单深入了解一下：

  • 条目（Entry）：

条目，在目录树中的每一个节点都是一个条目，通常对LDAP的添加、删除、更改、检索都是以条目为基本对象的。

  • 属性：

每个条目都可以有很多属性（Attribute），比如人有姓名、地址等属性。每个属性都有名称及对应的值，属性值可以有单个或多个，比如有多个邮箱。

  • 对象类（ObjectClass）：

对象类是属性的集合。

目录服务会通过树状结构存储数据，这样存储的优点就是查询效率高，例如：

树上的每一个节点都可以成为条目，每个条目都有很多属性，所以不同节点保存的数据可能不同，在LDAP中通过对象类（ObjectClass）来保存不同节点需要的属性。

每一个条目都有一个唯一的标识名（DN）。如上图的cn=test，它的DN就是

DN由多个RND(Relative Distinguished Name)通过逗号分割，如cn=test是RND，ou=managers也是一个RND。

上面涉及到了几个关键字。都简单说明一下，其实都是互联网命名组织架构使用的关键字：

  • DC：Domain Component。域名元件（部分），其格式是将完整的域名分为几部分，如域名example.com就变成dc=example,dc=com
  • UID：User Id。用户ID
  • OU：Organization Unit。组织单位，组织单位可以包含其他各种对象（包括其他组织单元）
  • CN：Common Name。公共名称
  • SN：Surname。姓
  • DN：Distinguished Name。标识名，由多个其他属性组成
  • RDN：Relative Distinguished Name。相对辨识名，类似于文件系统中的相对路径。
  • O：组织名

需要注意的是：dn必须是全局唯一的。

LDAP模型了解

LDAP的基本模型是建立在“条目”（Entry）的基础上。一个条目是一个或多个属性的集合，并且具有一个全局唯一的dn。ldap的基本模型其实也就是前面的目录树，可以回看前面的描述，不多说。

LDAP协议的其他点

LDAP的功能

在LDAP的功能模型中定义了一系列利用LDAP协议的操作，主要包含以下4部分：
查询操作：允许查询目录和取得数据，其查询性能比关系数据库好。
更新操作：目录的更新操作没关系数据库方便，更新性能较差，但也同样允许进行添加、删除、修改等操作。
复制操作：前面也提到过，LDAP是一种典型的分布式结构，提供复制操作，可将主服务器的数据的更新复制到设置的从服务器中。
认证和管理操作：允许客户端在目录中识别自己，并且能够控制一个会话的性质。

LDAP协议的特点

LDAP是一种目录服务，保存在特殊的数据库中，数据的读取速度远高于写入速度。
LDAP对查询做了优化，读取速度优于普通关系数据库。
LDAP不支持事务、不能进行回滚，需要进行这些操作的应用只有选择关系数据库。
LDAP采用服务器/客户端模式，服务器端用于存储数据，客户端提供操作目录信息树的工具，支持分布式结构。
LDAP中的条目以树形结构组织和存储。
LDAP基于Internet协议，直接运行在简单和通用的TCP/IP或其他可靠的传输协议层上，使连接的建立和包的处理简单、快捷，对于互联网和企业网应用都很方便。
LDAP协议简单，通过使用查找操作实现列表操作和读操作。
LDAP通过引用机制实现分布式访问，通过客户端API实现分布式操作（对于应用透明），平衡了负载。
LDAP实现具有低费用、易配置和易管理的特点，并提供了满足应用程序对目录服务所需求的特性。

LDAP的身份认证机制

  • 匿名认证：即不对用户进行认证，改方法仅对完全公开的方式适用。
  • 基本认证：通过用户名和密码进行身份识别，有分为简单密码和MD5密码认证。
  • SASL认证：即LDAP提供的在SSL和TLS安全通道基础上进行的身份认证，包括数字证书的认证。

如何使用

对于LDAP目录中保存的信息，可以使用LDIF(LDAP Interchange Format)格式来保存。这是一种标准文本文件格式，使用这种格式保存的LDAP服务器数据库中的数据可方便读取和修改。

LDIF文件常用来向目录导入和更改记录信息，这些信息需要按照LDAP中的schema的格式组织，并会接受schema的检查，不符合其要求的格式将会出现报错信息。

ObjectClass

在前面目录树也讲过，ObjectClass是属性的集合，不是很懂，细学一下。

在LDAP中，一个条目必须包含一个ObjectClass属性，且需要赋予至少一个值。每一个值将用作一条LDAP条目进行数据存储的模版，模版中包含了一个条目必须被赋值的属性和可选的属性。

objectClass有着严格的等级之分，最顶层是top和alias。例如，organizationalPerson这个objectClass就隶属于person，而person又隶属于top。

objectClass可分为以下3类：

  • 结构型（Structural）：如person和organizationUnit；
  • 辅助型（Auxiliary）：如extensibeObject；
  • 抽象型（Abstract）：如top，抽象型的objectClass不能直接使用。

对象类如上有三种类型：结构类型（Structural）、抽象类型(Abstract)和辅助类型（Auxiliary）。结构类型是最基本的类型，它规定了对象实体的基本属性，每个条目属于且仅属于一个结构型对象类。抽象类型可以是结构类型或其他抽象类型父类，它将对象属性中共性的部分组织在一起，称为其他类的模板，条目不能直接集成抽象型对象类。辅助类型规定了对象实体的扩展属性。每个条目至少有一个结构性对象类。

这样要好理解一些了，就是每个条目都需要顶一个objectclass，这样来表明这个条目记录的是什么信息，而objectclass中包含的一些属性有的是必须记录的，有的是可以无的，这样有便于目录信息的分布。

需要注意的是通过对象列可以方便的定义条目类型。每个条目可以直接继承多个对象类，这样就集成了各种属性。如果2个对象类中有相同的属性，则条目继承后只会保留1个属性。

在openLDAP的schema中定义了很多ObjectClass，列出几个常用的ObjectClass的名称：

  • account
  • alias
  • domain
  • organization
  • organizationalUnit
  • person

等。

属性（Attribute）

再简单提下属性，前面也是涉猎到了。属性类似于程序设计中的变量，可以被赋值。在openLDAP中声明了许多常用的Attribute。

每个条目都可以有很多属性（Attribute），比如常见的人都有姓名、地址、电话等属性。每个属性都有名称及对应的值，属性值可以有单个、多个，比如你有多个邮箱。

属性不是随便定义的，需要符合一定的规则，而这个规则可以通过schema制定。比如，如果一个entry没有包含在 inetorgperson 这个 schema 中的objectClass: inetOrgPerson，那么就不能为它指定employeeNumber属性，因为employeeNumber是在inetOrgPerson中定义的。

常见的属性名称及含义：

  • dc：domain component，域名的一部分

等。其实也就是前面目录树说过的。

并且前面也说了不同的ObjectClass，一般是具有一些必设属性和一些可选属性值。比如：objectclass=person，则必设属性为cn和sn，而可选属性为description、seeAlso等。

下面列出部分常用objectClass要求必设的属性：
● account：userid。
● organization：o。
● person：cn和sn。
● organizationalPerson：与person相同。
● organizationalRole：cn。
● organizationUnit：ou。

等

编写LDIF文件

对于LDAP目录中保存的信息，可以使用LDIF（LDAP Interchange Format）格式来保存。这是一种标准文本文件格式，使用这种格式保存在LDAP服务器数据库中的数据可方便读取和修改，这也是其他大多数服务配置文件所采取的格式。

初始状态下，LDAP是一个空目录，没有任何数据，可以使用OpenLDAP客户端工具ldapadd命令来完成添加数据的操作，该命令可将一个LDIF文件中的条目添加到目录。
